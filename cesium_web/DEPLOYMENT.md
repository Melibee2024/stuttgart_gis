# Cesium Web — Setup & Deployment Guide

This is the handoff doc for moving the Cesium part (backend + frontend) to another
machine (e.g. the university server). Read this first, then follow **Quick Start**.

> Context for whoever (or whichever Claude session) picks this up:
> the app was developed against a local "dummy" `3DCity` Postgres DB. The target
> university machine runs Postgres **on the same machine** as the backend, and the
> DB is **already loaded** with the same schemas/data. So this is essentially
> reproducing the local dev setup on that machine.

---

## Architecture

- **Backend** — `stuttgart-backend/` — Express (`server.mjs`, ESM, Node).
  - Connects to Postgres via `pg`, reads credentials from `.env` (dotenv).
  - Serves the REST API (`/api/...`) plus static `/tiles` (pg2b3dm output from
    `public/tiles`) and `/media`.
  - `POST /api/regenerate-tiles` shells out to the `pg2b3dm` CLI.
- **Frontend** — `stuttgart-digital-twin/` — Vite + CesiumJS (Cesium loaded via CDN
  in `index.html`, not bundled). Single-page app in `src/main.js`.
- **Tiles** — generated by `pg2b3dm` (.NET global tool) from the DB view
  `nexus3d.v_ifc_tiles`. Output lands in `stuttgart-backend/public/tiles/`
  (gitignored — regenerate per machine).
- **Two 3D tilesets in the scene:**
  1. `ifcTileset` — the IFC building (pg2b3dm output, served by our backend).
  2. `baseTileset` — citydb LoD2, **Cesium ion asset 96188** (needs an ion token).

### DB objects the backend depends on
- `nexus3d.building_georef`, `nexus3d.ifc_elements`, `nexus3d.v_ifc_tiles`
- `citydb.feature`, `citydb.geometry_data`, `citydb.property`
- `public.data_fusion_view`, `qfield_data.building_photos`
- **PostGIS** must be enabled.

## Quick Start (target machine)

### Prerequisites
- **Node.js** (v20+; developed on v24).
- **.NET `pg2b3dm` tool**: `dotnet tool install -g pg2b3dm`
  (only needed to (re)generate tiles; alternatively copy a `public/tiles/` folder over).
- Postgres with the `3DCity` DB + PostGIS already present.

### 1. Get the code
```
git clone <repo>
cd <repo>/cesium_web
```
`node_modules/`, `.env`, and `public/tiles/` are gitignored — recreate them below.

### 2. Backend
```
cd stuttgart-backend
npm ci
copy .env.example .env      # Windows (or: cp .env.example .env)
```
Edit `.env` and set the **real DB credentials for this machine**:
```
DB_USER=postgres
DB_HOST=localhost
DB_NAME=3DCity
DB_PASSWORD=<this machine's postgres password>
DB_PORT=5432
PORT=5000
```
> ⚠️ **Most likely failure point:** the server's Postgres password / DB name differ
> from the dev machine. The backend prints a clear error and exits if `DB_PASSWORD`
> is unset.

### 3. Generate the IFC tiles
With the backend `.env` filled in and `pg2b3dm` installed:
```
pg2b3dm --connection "Host=localhost;Database=3DCity;Username=postgres;Password=<pw>;Port=5432" -t nexus3d.v_ifc_tiles -c geom -a "global_id,ifc_class,name,storey,z_min_ellipsoidal,z_max_ellipsoidal,element_height_m" -o ./public/tiles
```
(Or start the backend and call `POST /api/regenerate-tiles` once.)
Result: `public/tiles/tileset.json` + `content/*.glb` + `subtrees/`.

### 4. Frontend
```
cd ../stuttgart-digital-twin
npm ci
copy .env.example .env      # then edit
```
In `.env`:
- `VITE_API_BASE` — leave as `http://localhost:5000` if you open the app in a
  browser **on this machine**. If reaching it from another computer on the LAN,
  set it to `http://<server-hostname-or-ip>:5000`.
- `VITE_ION_TOKEN` — your Cesium ion token (https://ion.cesium.com/tokens).
  Without it the citydb base tileset uses Cesium's rate-limited default token
  (works but shows a warning banner and may stop loading).

### 5. Run
- Backend: `npm start` (in `stuttgart-backend/`)
- Frontend: `npm run dev` (in `stuttgart-digital-twin/`) → http://localhost:5173
  - or production: `npm run build` then serve the `dist/` folder.

### 6. Verify
- `GET http://localhost:5000/api/tiles/status` → `{"running":false}`
- `GET http://localhost:5000/api/buildings/filters` → IFC classes + storeys
- `GET http://localhost:5000/api/georef` → a building record with heights/footprint
- Frontend: the IFC building renders; clicking it fills the right-hand info panel;
  the "Hide citydb buildings" toggle hides the whole citydb base layer.

## What was already fixed in this branch (history / don't re-debug)
- **Backend wouldn't start:** `dotenv` was missing from `node_modules` →
  `npm install` added it. `package.json` `main` pointed at a non-existent
  `index.js`; added `start` / `dev` scripts.
- **No `.env`:** created locally (gitignored). Use `.env.example` as the template.
- **IFC "missing":** `public/tiles` had never been generated → ran `pg2b3dm`.
- **Grey-box "artifact":** it was a neighboring **citydb LoD2 building** (base
  tileset), not IFC geometry. The "Hide citydb building under IFC" toggle only
  clipped the single matched twin. Changed (per request) so the toggle now hides
  the **entire** citydb base tileset, and the footprint clip is **always on** (the
  IFC's own LoD2 twin is permanently removed to avoid z-fighting). Label is now
  "Hide citydb buildings".
- **Deployment-readiness:** `src/main.js` now reads `VITE_API_BASE` and
  `VITE_ION_TOKEN` from env instead of hardcoding `localhost:5000` / the default
  ion token.

## Optional hardening (only if exposed beyond LAN)
- Restrict CORS in `server.mjs` (currently open `cors()`).
- Run the backend under PM2 or as a Windows service for auto-restart.
- Put nginx in front: serve the built frontend + reverse-proxy `/api`, `/tiles`,
  `/media` to the backend → single origin, no CORS/mixed-content issues.
- Use HTTPS (a TLS cert) if reachable from the public internet.
